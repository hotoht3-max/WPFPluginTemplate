# Архитектура и Стандарты Разработки: Сквозная Колонна RAM BIM

Настоящий документ определяет жесткие архитектурные рамки, паттерны проектирования и правила модификации плагина «Сквозная колонна RAM» для платформы Tekla Structures 2022. Документ является обязательным руководством для разработчиков и внешних ИИ-агентов. Любые отклонения от описанной структуры являются дефектом архитектуры.

---

## 1. Компонентная Модель Данных и Пайплайн (Data Flow)

Плагин использует строгое разделение между плоскими данными диалогового окна Tekla и строго типизированной объектной моделью бизнес-логики.

### 1.1 Роль класса `PartSettings.cs`
Класс `PartSettings` является универсальным неделимым контейнером для параметров любой физической детали Tekla (Балка, Ветвь, Надколонник, Пластина). Запрещено передавать в методы сборки геометрии разрозненные строки профилей или материалов. Атрибуты детали всегда инкапсулируются внутри `PartSettings`:
- Свойства профиля и материала (`Profile`, `Material`).
- Нумерация и идентификация (`PartPrefix`, `PartStartNo`, `AssemblyPrefix`, `AssemblyStartNo`).
- Визуальные и системные свойства (`Name`, `Class`, `Uda`).

### 1.2 Сквозной пайплайн передачи данных
Передача данных от интерфейса к модели происходит строго по следующей цепочке:
1. **WPF View (`MainWindow.xaml` / `UIControls`)**: Биндинг элементов интерфейса к свойствам ViewModel.
2. **`MainWindowViewModel.cs`**: Плоские свойства с атрибутами `[StructuresDialog("FieldName", typeof(TD.Type))]`.
3. **`PluginData.cs`**: Плоские зеркальные поля с атрибутами `[StructuresField("FieldName")]`, считываемые ядром Tekla.
4. **`Models/BuiltUpColumnData.cs`**: Чистый C# класс объектной модели. На этом уровне плоские поля атрибутов деталей заменяются на экземпляры `PartSettings`.
5. **`Services/PluginDataMapper.cs`**: Единственная точка сборки. Маппер обязан трансформировать плоский список из `PluginData` в объектную структуру `BuiltUpColumnData` исключительно через внутренний метод `CreatePart`.

⚠️ **Чеклист при добавлении нового параметра.** Обязательно обновить все 4 точки:
1. `PluginData.cs` — поле с `[StructuresField]`
2. `MainWindowViewModel.cs` — свойство с `[StructuresDialog]`
3. `BuiltUpColumnData.cs` — свойство или вложенный объект
4. `PluginDataMapper.cs` — строка маппинга в `CreatePart` или прямое присвоение

Пропуск любого пункта приводит к тихому багу: параметр не сохраняется или не передаётся в построение. Компилятор ошибку не покажет.

*Пример маппинга новой детали:*
```csharp
HeadBeam = CreatePart(data.HB_Profile, data.HB_Material, data.HB_PartPref, data.HB_PartNo, data.HB_AssyPref, data.HB_AssyNo, data.HB_Name, data.HB_Class, data.HB_UDA)

```

## 2. Изоляция Сервисов и работа с Tekla API

Бизнес-логика плагина не имеет права напрямую вызывать конструкторы системных объектов Tekla, кроме как через специализированные изолированные сервисы.

### 2.1 Использование `TeklaProfileHelper.cs` (Геометрия Сечений)

Запрещено использовать строковый парсинг имен профилей (сортамента) для вычисления высоты или ширины деталей в КМД-плагинах. Из-за разницы фактических и номинальных размеров ГОСТ-сортамента это приводит к коллизиям.

* **Паттерн запроса габаритов:** Для получения точных размеров используется метод `TeklaProfileHelper.GetActualDimensions`.
* **Жизненный цикл:** Чтобы узнать габариты, деталь сначала собирается по координатам базовой оси и вставляется в модель (`.Insert()`). Затем объект передается в хелпер, который вытаскивает реальные габариты сечения из актуальной базы модели с учетом угла поворота. На основе полученных точных данных вычисляются векторы эксцентриситетов и ось детали сдвигается через `.Modify()`.

### 2.2 Роль `TeklaPartBuilder.cs`

Класс отвечает за сборку объектов Tekla в оперативной памяти. Метод `CreateBranch` принимает точки и объект `PartSettings`, инициализирует нативную деталь, задает префиксы и базовый поворот.

* **Важное правило:** `TeklaPartBuilder` никогда не вызывает метод `.Insert()`. Он возвращает "чистый" объект в памяти. Окончательную фиксацию детали в базе данных модели осуществляет только фабричный метод в оркестраторе.

### 2.3 Роль `TeklaComponentService.cs` и Борьба с Автостандартами

Сервис инкапсулирует вставку узлов (`Connection`) и деталей (`Detail`).

* **Правило Двойного Хука (Double Hook):** При вставке любого компонента Tekла API приоритетно применяет системные AutoDefaults, затирая настройки плагина. Для подавления этого поведения сервис обязан применять паттерн:
1. `component.Insert()` (создание объекта).
2. Повторный вызов `component.LoadAttributesFromFile(presetName)`.
3. Обнуление числовых корней (передача `0` вместо строк в атрибуты вроде `ac_root`).
4. Вызов `component.Modify()` для принудительной записи данных поверх автостандартов.

**Причина обязательности паттерна:** после вызова `component.Insert()` Tekla API автоматически применяет системные AutoDefaults, перезаписывая все пользовательские настройки пресета. Без повторного `LoadAttributesFromFile` + `Modify()` пользовательские настройки будут проигнорированы. Убирать или сокращать паттерн запрещено даже при видимой «оптимизации».

## 3. Разделение ответственности: Оркестратор vs Математика

Код построения геометрии разделен по принципу SOLID на два ключевых класса:

1. **`ModelPlugin.cs` (Оркестратор / Контроллер)**: Метод `Run` принимает точки вставки, распределяет потоки данных, вызывает сервисы и фабричные методы (`BuildSupColumn`, `BuildHeadBeam`). Он отвечает за транзакции и запись в БД модели. В оркестраторе запрещены сложные тригонометрические расчеты шагов раскосов или уровней пластин.
2. **`Services/ColumnGeometryBuilder.cs` (Чистый математический движок)**: Вычисляет массивы точек, локальные координатные системы, векторы сдвигов, списки Z-отметок для решетки, планок и диафрагм.
* *Ограничение:* Класс является "чистым". В нем запрещено подключать пространства имен `Tekla.Structures.Model` и создавать физические объекты. Он оперирует только базовыми геометрическими типами (`Point`, `Vector`, `Line`).

## 4. Экосистема Интерфейса (UIControls)

Интерфейс плагина построен на принципах атомарного дизайна и жесткой модульности. Главное окно `MainWindow.xaml` является лишь верхнеуровневым контейнером-маршрутизатором.

### 4.1 Повторное использование (Atomic UI Controls)

Все однотипные строки ввода вынесены в кастомные пользовательские элементы со встроенными `DependencyProperty`:

* `PartAttributesRowControl.xaml` — Единый блок атрибутов детали (Профиль, Материал, Префиксы, Нумерация, Класс, UDA). Обязателен к передаче текстового биндинга и имени атрибута для корректной работы макросов-галочек.
* `PositioningRowControl.xaml` — Универсальный блок пространственного позиционирования деталей (На плоскости, Поворот, По глубине и их смещения).
* `ComponentSelectorControl.xaml` — Поля выбора номера макроса Tekla и файла его настроек (Пресет).

### 4.2 Жесткие правила координационной сетки (Grid Alignment)

Чтобы элементы интерфейса на разных вкладках и внутри вложенных экспандеров не "плясали" по вертикали, на всех уровнях вложенности должна соблюдаться единая координационная сетка столбцов `Grid.ColumnDefinitions`:

1. **Столбец 0 (`Width="Auto" MinWidth="25"`)** — Строго под чекбоксы макросов-галочек (`WpfFilterCheckBox`).
2. **Столбец 1 (`Width="200"`)** — Строго под текстовые подписи и лейблы (`Label`). Внутри элементов управления `VerticalAlignment` всегда должен быть равен `Center`, а высота строки ввода фиксируется на уровне `Height="24"`.
3. **Столбец 2 (`Width="*"`)** — Поля ввода, выпадающие списки и вложенные селекторы компонентов, растягивающиеся на всю доступную ширину.

### 4.3 Выравнивание составных контролов

Если составной контрол (например, `PositioningRowControl`) имеет внутреннее смещение под свой текстовый лейбл (фиксированные 145px), то при вызове его в сторонних контейнерах, где присутствует мастер-галочка, контейнер обязан компенсировать разницу внешним отступом или дополнительным столбцом (например, столбец `Width="80"` под галочку + `145px` внутреннего лейбла = `225px`), чтобы выровнять выпадающие списки пиксель в пиксель с основной сеткой окна.

## 5. Законы генерации кода для ИИ (AI Instructions)

1. **Запрет на частичный код:** При изменении логики внутри методов `ModelPlugin.cs` или `ColumnGeometryBuilder.cs` ИИ обязан выводить модифицированный метод целиком. Использование комментариев вида `// Здесь старый код...` или заглушек запрещено.
2. **Синтаксический парсинг:** Любое новое текстовое поле, принимающее массивы чисел, шаги или диапазоны, обязано обрабатываться исключительно через `StringParserService`.
3. **Строгая типизация углов:** При передаче углов поворота в методы нативного API Tekla всегда четко разграничивать радианы и градусы. Названия переменных углов должны содержать суффиксы `Deg` или `Rad`.
4. **Конвенция префиксов:** Все новые переменные в плагине обязаны подчиняться сквозным системным префиксам:
* `B_` — Ветви колонны (Branch)
* `L_` — Решетка (Lacing)
* `S_` — Планки (Strut)
* `D_` — Диафрагмы (Diaphragm)
* `NK_` — Надколонник (SupColumn)
* `HB_` — Балка оголовка (HeadBeam)
5. **Запрет мутации входных данных:** Сервисные методы не имеют права изменять поля объекта `BuiltUpColumnData`, переданного как входной параметр. Если метод должен модифицировать `PartSettings` (например, вычислить профиль пластины по геометрии), он обязан создать локальную копию объекта `PartSettings` и работать с ней. Оригинальный `colData` после вызова метода должен остаться в исходном состоянии.
6. **Запрет `out`-параметров:** Если метод возвращает более одного значения — использовать именованный вспомогательный класс результата (в рамках .NET Framework 4.8) или `ValueTuple`. `out`-параметры внутри наших собственных методов запрещены как нечитаемый антипаттерн, опасный при рефакторинге (исключение — вызовы нативного API Tekla).
7. **Разделители секций в коде:** Для визуальных разделителей внутри методов использовать `// ---`, не `/// ---`. Тройной слэш (`///`) зарезервирован исключительно для XML-документации публичных членов класса; внутри тел методов он не имеет смысла и конфликтует с компилятором при включённой генерации документации.
